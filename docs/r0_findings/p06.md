## Construction plan for the IML (§8.7), enabled by these findings

> **SUPERSEDED (2026-07-09) — offset distances only.** The offset SEQUENCE
> derivation below (`face_mm`, then `core_mm/2`) is wrong: it optimized for
> making the TOTAL two-wall consumption equal the P0 `stack_mm` formula,
> which silently deleted the panel's INNER face sheet (a sandwich panel is
> outer face / core / inner face PER WALL — product review caught the
> missing layer). The correct chain is full-value (`face_mm`, `core_mm`,
> `face_mm`); `stack_mm` is the PER-WALL panel thickness, and total
> consumption is `2*stack_mm`. See "Addendum: 3-layer panel correction"
> below and `probe_ocp_offset_3layer.py`'s results. Everything else in this
> section (API, wire/loft/boolean route, per-station sourcing, false-spar
> note) still stands.

**The critical question this probe resolved**: `cq.Wire.offset2D(-d, kind="intersection")`
is a real, working API (no raw-OCP fallback needed) — but a SINGLE whole-loop
offset by distance `d` doesn't remove `d` of local wall thickness, it removes
`2d` (confirmed empirically to within 0.007mm on a real section: measured
ratio 2.00x). Offsetting the WHOLE closed airfoil loop inward moves the upper
AND lower walls inward *simultaneously* in one pass, so the local
upper-to-lower gap shrinks by `2d`, not `d`.

This matters because `backend/schema/validators.py`'s P0 check compares
`stack_mm = core.thickness_mm + 2*face_sheet_mm` (**one** core factor, not
two) against local thickness — already a FROZEN, passing check for every
existing device config, including the tightest margin in the whole battery
(`te_half_twisted_moderate.yaml`: face=0.4mm, core=2.0mm at its 190mm tip
chord, min thickness 5.501mm). A naive "offset by face_mm, then by core_mm"
two-pass construction would consume `2*face_mm + 2*core_mm` — MORE than
`stack_mm` — which would silently invalidate that frozen config's geometry
even though it correctly passes P0 validation (a real, easy-to-miss bug this
probe caught before any implementation code existed).

The offset sequence whose consumption exactly equals `stack_mm` (so it's
guaranteed safe for every config P0 already accepts, with zero changes to
`tolerances.py` or the validator):

```
face_sheet_IML = OML_wire.offset2D(-face_sheet_mm)        # consumes 2*face_sheet_mm
hollow_IML     = face_sheet_IML.offset2D(-core_mm / 2.0)  # consumes 2*(core_mm/2) = core_mm
# total consumed = 2*face_sheet_mm + core_mm == stack_mm, exactly
```

Verified both analytically and empirically on `te_half_twisted_moderate.yaml`'s
real numbers above (measured 18.022mm vs analytically-predicted 18.017mm —
agreement to 0.03%, the residual being offset-algorithm/polygon-discretization
noise, not a modeling error).

**Per-body, per-layer construction** (mirrors P2/P4's own loft pattern exactly
— per-station polygon wires, `cq.Solid.makeLoft(wires, ruled=True)`, never a
spline or a literal OCC primitive):

1. At each of the same `N_STATIONS` sections `analytic_section_points`
   already uses (or the same station set P2's OML loft uses, for bodies where
   no analytic re-sectioning is available — see Implementation notes below),
   build the per-station **face_sheet_IML** and **hollow_IML** wires via the
   chained offset above.
2. Loft each wire family (`ruled=True`) into `face_sheet_iml_solid` and
   `hollow_iml_solid`.
3. `face_sheet_shell = fuzzy_cut(body, face_sheet_iml_solid)` — the face-sheet
   layer.
4. `core_shell = fuzzy_cut(face_sheet_iml_solid, hollow_iml_solid)` — the core
   layer, sitting strictly between the two IMLs.
5. `hollow_iml_solid` itself is the true interior cavity — ribs/spars/hardware
   are built inside it, never inside `body` directly.

**Ramped drop-offs** (`ramp_ratio`, D11): not a new boundary — a ramp is just
the SAME construction with `core_mm` linearly interpolated to 0 over a
`ramp_ratio * core.thickness_mm`-long spanwise/chordwise band near an edge,
hinge land, or hardpoint (i.e., `hollow_IML` and `face_sheet_IML` converge
back toward each other, and toward the OML, over the ramp band) — reuses the
identical per-station offset machinery with a station-varying `core_mm`
rather than a constant, so no separate construction path is needed.

**Bodies this applies to**: both P4 outputs (`wing`, `control_surface`) —
neither is the raw P2 OML, so per-station analytic sectioning
(`cove_profile.py`'s `analytic_section_points`) does NOT apply directly to
either (it assumes the untouched ruled-loft OML). P6's per-station sections
for the sandwich construction are instead taken directly from
`build_planform_sections`'s own station polygons (unaffected by the TE cut,
since the cut only removes material near the hinge/aft-of-hinge — the
per-station OUTER LOOP used for offsetting is the ORIGINAL airfoil section,
not a re-derived cross-section of the already-cut body). The face-sheet/core
lofts are built from these ORIGINAL per-station sections (same as the OML's
own loft input), then cut against each ACTUAL body (`wing` or
`control_surface`) rather than against the OML — `fuzzy_cut(wing,
face_sheet_iml_solid)` etc.

**This does NOT automatically close the device-cut face correctly** — at the
TE cut plane, `wing`/`control_surface` have a boolean-cut face that isn't
part of the original closed airfoil loop the per-station offset assumes, so
subtracting a solid built from the UNCUT airfoil's sections there would treat
the cove/cut boundary as if it were still solid airfoil material beyond it.
This is exactly why plan.md §8.7 lists "false spars close device cut faces"
as an explicit, separate construction step, not a side effect of the IML
subtraction — the false spar is a flat closing wall built directly ON the cut
plane (a face-sheet+core sandwich cap, same layer stack as the rest of the
skin) that gives the hollow interior a real boundary there. Implementation
order: build the "clean" IML/core/hollow construction first (this section),
then add the false-spar closing wall at each device cut face as its own
step — don't try to make the per-station offset "handle" the cut boundary
implicitly.

**Deliberately NOT done here**: a smooth/spline offset, or `BRepOffsetAPI`
called on a FACE (rather than the per-station wire route) — both are
unverified for this project's dense-polygon convention and would risk exactly
the "unverified boundary" trap F13 warns about. `cq.Wire.offset2D` on the same
polygon wires the OML already uses is the smallest change consistent with
every prior phase's construction style.

## Addendum: body-restricted core + camber-line upper/lower split (post-viewer review)

Two defects/gaps in the first clean-span implementation, both caught by
LOOKING at the rendered output in the dev viewer (exactly what that tool
exists for), fixed in `backend/geometry/iml.py`:

1. **Core band was never restricted to the body.** `face_sheet_shell =
   body − face_IML` inherits the device cut by construction, but the core
   was built as `face_IML − hollow_IML` alone — pure loft algebra, no body
   term — so it sailed uncut through the TE control-surface pocket.
   Fix: `core_shell = (face_IML − hollow_IML) ∩ body`. One extra boolean
   (`core_body_common_s` in timings).

2. **Skins are now delivered split into upper and lower shells** (product
   requirement; matches molded-composite reality — one skin per mold half —
   and the later per-surface midsurfaces for Ansys). Split mechanism: a
   "below-camber" parting prism, built from already-probed machinery only
   (polygon wires → ruled loft → fuzzy booleans, no new OCC boundary):
   - The camber polyline per station is EXACT by midpointing paired placed
     points: `resample.py` guarantees upper/lower surfaces share one cosine
     x-grid with a shared LE (`pts[i]` upper ↔ `pts[N-1-i]` lower at equal
     x), and placement is affine per-point, so midpoints commute with it.
   - Extended past LE/TE by `PARTING_EXTENSION_CHORD_FRAC` (5% chord) so the
     prism crosses the skin's LE curve and blunt-TE closing wall
     transversally instead of ending ON them (F4-style coincidence hazard
     otherwise); closed downward to a floor 0.5 chord below the section;
     lofted ruled across stations (planar per station — placed sections sit
     at constant Y).
   - `lower = ring ∩ prism`, `upper = ring − prism`. The prism's camber
     surface passes through the hollow interior mid-chord (no material →
     cuts nothing) and only meets skin material near LE/TE, transversally.
   - PROVISIONAL parting definition: the real mold parting curve (max
     half-breadth per station) arrives with P15/P16 and supersedes this for
     tooling; the camber split is the right first cut for skin-body
     separation and per-surface FEA.

Verification is folded into the export path itself (no extra booleans —
volume sums over already-computed shapes): zero shards (F3), every kept
solid watertight, and upper+lower exactly partitioning each ring (<0.1%
volume deviation) — asserted on every export run, both configs.

## Addendum: 3-layer panel correction (product review, 2026-07-09)

A sandwich panel is **three layers per wall** — outer face sheet / core /
inner face sheet. The first implementation delivered only two (outer face +
half-thickness core, no inner face sheet), because the original offset
derivation above treated the FROZEN P0 formula `stack_mm = core + 2*face` as
a budget for the TOTAL two-wall consumption and chose offsets to match it.
That was a misreading: `stack_mm` is the PER-WALL panel thickness (exactly
what the P3 hinge-axis-margin gate already uses it as), and the P0 F15 check
(`stack_mm <= 0.8 * min sampled local thickness`) is a per-wall sanity
check, not a two-wall feasibility guarantee.

Corrected construction (`backend/geometry/iml.py`):

```
face_IML   = OML_wire.offset2D(-face_mm)   # outer face sheet, wall = face_mm
core_IML   = face_IML.offset2D(-core_mm)   # core, wall = core_mm
hollow_IML = core_IML.offset2D(-face_mm)   # inner face sheet, wall = face_mm
```

Rings (each body-restricted, each split upper/lower by the parting prism):
`face_outer = body − face_IML`, `core = (face_IML − core_IML) ∩ body`,
`face_inner = (core_IML − hollow_IML) ∩ body`, cavity = `body ∩ hollow_IML`.

**Consequence, deliberately accepted (no frozen gate/config touched):**
total local consumption is now `2*stack_mm`, and several frozen configs are
thinner than that near the aft tip (audited analytically with the same
helper P0 uses, min sampled at x/c=0.9): te_half_twisted_moderate 5.6 vs
5.501mm (walls just merge), te_half 7.6 vs 8.69mm (fits), devices_full 7.6
vs 5.21mm and high_taper 1.8 vs 1.16mm (over-packed aft). Where they don't
fit, the innermost offsets self-clip (`kind="intersection"`) and the hollow
locally vanishes — walls merge into solid laminate, which is also roughly
what real construction does aft of the rear spar. probe_ocp_offset_3layer.py
(results above) verified on the real kernel that clipped wires stay single/
valid/closed, lofts against unclipped stations stay valid, rings stay
shard-free, and volume conservation is exact (delta 0.00). The proper
physical treatment of these regions is ramped core drop-offs (D11) and the
P6 gate's IML audit — tracked, not silent.

## Addendum: cove-arc IML fidelity (wing body, 2026-07-11)

Resolves the "DELIBERATE SCOPE LIMIT" this file's construction plan flagged
above: inside an enabled te_surface's spanwise window, `wing`'s actual outer
boundary near the hinge is the cove arc (`te_cut.py`'s `cove_region`,
radius `R + COVE_CLEARANCE_MM` from the station's hinge point C —
cove_profile.py), not the plain airfoil skin the per-station offset chain
was still using. Concretely: `face_iml_solid` there was the inward offset
of the ORIGINAL uncut airfoil — a solid whose boundary near the hinge sits
close to the un-notched skin, radius comfortably larger than the true cove
radius. `face_outer_shell = body − face_iml_solid` at those stations was
therefore either EMPTY (the cove notch cuts deeper than face_mm from the
original skin — the common case, since `body`'s true boundary sits at
`R+COVE_CLEARANCE_MM`, typically well inside where the uncut-skin offset
puts `face_iml_solid`'s boundary) or an uncontrolled sliver, never a real
face_mm-thick layer measured from the true surface — exactly where the CS
nose bears against the cove lip and P6's gate pass criterion ("IML audit:
min wall ≥ face-sheet stack everywhere, sampled") would have caught it.

**No new R0 probe** — no new third-party boundary. The fix reuses
cove_profile.py's own concentric-arc construction (already probed/verified
for the nose/cove arcs themselves) at a larger radius, which is why it
carries no self-intersection risk the existing `offset2D`-based chain has
to guard against (a concentric arc grown by `d` is trivially still a simple
arc — no clipping, no `offset2D` call at all): `cove_profile.build_cove_arc_points`
gained an `extra_radius_mm` parameter (default 0.0, every existing call
site unaffected); `te_cut.build_cove_offset_region(config, sections, d)`
lofts that arc over the EXACT same station set `cove_region` itself uses
(N_STATIONS=24, full `[0, axis_len]` span, same feet/tangency machinery) at
radius `R + COVE_CLEARANCE_MM + d`; `iml.build_sandwich_lofts` subtracts
this from each of the three offset lofts (`d = face_mm`, `face_mm+core_mm`,
`stack_mm`) when `te_surface` is enabled, before the existing
`build_sandwich_body` boolean chain runs unchanged. Set algebra: annulus
between two of these cove-radii, intersected downstream with `body`
(already cut to the true `R+COVE_CLEARANCE_MM` boundary by te_cut.py), is
exactly a face_mm- or core_mm-thick layer measured from the TRUE cut
surface — same reasoning as the plain clean-span case, just re-centered on
the hinge point instead of offset from the local skin curve.

**Verified against the real kernel** (`tests/configs/devices/te_half.yaml`,
wingo.coder, 2026-07-11): sandwich shells still 3-ring-partitioned,
watertight, shard-free (volumes shifted slightly from the pre-fix run, as
expected — only the cove-window portion of each ring changed). New cost:
`cove_fidelity_s=93.9s` on top of the existing lofts. `false_spar.py`
consumes this same `hollow_iml_solid`, so it was re-verified in the same
run, not assumed: volume dropped 31442→10237 mm³ and CS clearance improved
5.50→7.42mm, consistent with the cavity now correctly stopping at the true
cove boundary instead of the old over-extended (uncut-airfoil-based) one.

**Deferred, not forgotten**: the control surface's own nose sandwich (mirror
construction, offset INWARD from a radius-R arc rather than outward from
`R+COVE_CLEARANCE_MM`) is not built here — CS sandwich shells aren't wired
into any export/gate path yet.

## Addendum: ramped drop-offs (D11, wingtip edge, 2026-07-11)

**First attempt was wrong, caught empirically, not by inspection.** Making
`core_mm` a per-station value (nominal, tapering to ~0 within
`ramp_ratio*core_mm` of the tip) is necessary but NOT sufficient:
`cq.Solid.makeLoft(..., ruled=True)` linearly blends the 2D offset WIRE
between consecutive INPUT stations — it has no notion of the per-station
`core_mm` function in between. `tests/configs/devices/te_half.yaml` declares
only 2 planform stations (root, tip, span 1200mm); with `core_mm` computed
only at those two points, the loft blended from full core (root) to
near-zero core (tip) across the ENTIRE 1200mm span, not the intended 9mm
(`ramp_ratio=3.0 × core_mm=3.0mm`). Measured effect: `core_ring` volume
dropped ~43% (2330387.6→1338694.7 mm³) — an order of magnitude more than a
9mm-of-1200mm taper should ever remove. Caught by sanity-checking the
verified-run volume against the config's own `span_mm`/`core.thickness_mm`/
`ramp_ratio` before treating the (numerically "valid": watertight,
partitioned, shard-free) result as correct — the F3/watertight/partition
asserts this module already runs cannot catch a construction that's VALID
but represents the WRONG shape.

**Fix**: `iml._insert_ramp_station` inserts one extra station AT the ramp
boundary (`y_tip − ramp_len_mm`) into a local copy of the section list used
only for the face/core/hollow offset+loft construction (not the OML or
parting prism, which keep using the original station list unmodified) —
built via the exact same `interp_station`+`place_section` pipeline
`build_planform_sections` itself uses, so it's geometrically identical to
what a declared station there would have produced. This pins the loft
segment boundary at the ramp start, confining the blend to the intended
zone regardless of how sparse the declared planform stations are.

**Re-verified against the real kernel** (same config, wingo.coder,
2026-07-11): `core_ring` now 2322667.5 mm³, a ~0.33% reduction from the
pre-ramp cove-fidelity baseline (2330387.6 mm³) — consistent with tapering
only the last 9mm of a 1200mm span. Sandwich still 3-ring-partitioned,
watertight, shard-free; false spar unaffected (vol/CS-clearance essentially
identical, as expected — nowhere near the tip).

**Deferred, not forgotten** (plan.md §8.7 lists 4 ramp locations — only
"edges"/wingtip is built here): hinge-land ramping needs hinge attachment
points (P7); joint ramping needs joint definitions (P11); hardpoint ramping
needs hardware pockets (P8). The cove-fidelity wedge (previous addendum) is
sized off the NOMINAL `core_mm`, not the ramped value — a device window
whose outboard edge falls inside the tip's ramp zone is an untested
interaction between these two corrections (not expected on any current
config; not yet verified if it happens).

## Addendum: ramp regression on high_taper.yaml — two real bugs, both fixed (2026-07-11)

Testing the ramp against `tests/configs/edge/high_taper.yaml` (10:1 taper,
`mirror:true`, single-ply 0.2mm face + 0.5mm core — explicitly a
self-intersection stress-test config per its own file comment) surfaced a
genuine crash: `build_sandwich_body`'s `face_inner_upper =
fuzzy_cut(face_inner_shell, parting_solid)` raised `RuntimeError:
BRepAlgoAPI_Cut failed`. Isolated with `git show <pre-ramp-commit>:iml.py`
swapped onto the workspace: the pre-ramp code succeeded on this exact
config, confirming the ramp work was the cause, not a pre-existing latent
bug. Two independent, compounding defects, found by tracing every
intermediate shape's validity through `build_sandwich_body`'s exact boolean
chain (a standalone script replicating it line-by-line with
`isValid()`/`Solids()`/volume checks after each step — the project's
existing `isValid()`-is-not-sufficient lesson, r0_findings/p02.md, applies
here too: every intermediate WAS reported valid, the crash only showed up
at the actual boolean call):

1. **`RAMP_MIN_CORE_MM=0.01` accidentally equaled `KERNEL_TOLERANCE_MM`
   exactly.** The tip station's core layer thickness was floored to the
   same value as the kernel's own numerical precision — building a
   geometric feature AT a tolerance boundary is asking for exactly this
   kind of fragility. Fix: moved the constant into `backend/tolerances.py`
   as `RAMP_MIN_CORE_MM = 0.1` (10x `KERNEL_TOLERANCE_MM`, same "10x/50x
   comfortable margin" reasoning `FALSE_SPAR_COVE_STANDOFF_MM` already
   established) — this alone reduced the failure's severity (28 degenerate
   micro-fragments at a larger fuzzy value down to 2) but did NOT fully fix
   it.
2. **The real cause: `parting_solid` was still lofted from the ORIGINAL
   (unexpanded) `sections`, while `face_inner_shell` (and every other ring)
   derived from lofts built over `ramp_sections` (the ramp-boundary station
   inserted by `_insert_ramp_station`).** Two lofts with DIFFERENT
   per-station Y-sampling, boolean'd against each other, is a topology
   mismatch this project's `_loft_region` precedent (te_cut.py) already
   warns about WITHIN a single loft's stations — this is the same lesson
   one level up, across separate lofts that get cut/intersected together.
   `face_inner_lower` (the COMMON half) succeeded regardless; only
   `face_inner_upper` (the CUT half) failed, and failed identically at
   every fuzzy value tried (0.001 default, 0.01, 0.05, 0.1, 0.5) — a sign
   of a structural mismatch, not a tolerance-tunable one. Fix: `parting_solid`
   is now lofted from `ramp_sections` too (`build_sandwich_lofts`) — every
   loft that participates in the same boolean chain shares one station set.
   The OML loft (built earlier, in loft.py, from the original `sections`)
   is the one deliberate exception — it's unaffected by ramping and OCC
   booleans a solid against a solid regardless of the OTHER solid's own
   internal loft parameterization, so this exception is safe.

**Re-verified against the real kernel** (`high_taper.yaml`, wingo.coder):
full `build_sandwich_lofts`/`build_sandwich_body` chain now succeeds
end-to-end, every intermediate shape valid + watertight + single-solid,
`face_inner_lower(44991.8) + face_inner_upper(46099.4) ≈
face_inner_shell(91091.4)` (partition sums correctly). Also re-verified
`te_half.yaml` still passes (this fix touches shared code every config
goes through).

**Lesson for future construction work in this module**: whenever a new
per-station correction (ramping, cove-fidelity, or anything future) adds or
removes stations from ONE of the lofts that feeds `build_sandwich_body`'s
boolean chain, every OTHER loft that gets boolean'd against it must use the
same expanded/modified station list — a silent topology mismatch can pass
every per-shape `isValid()` check and still make the boolean itself fail
(or worse, silently succeed with subtly wrong geometry) on a sufficiently
extreme config, and won't show up on gentler configs like `te_half.yaml`.

## Addendum: ribs (plan.md §8.7, "plane ∩ inner volume") — R0 trail (2026-07-11/12)

`backend/geometry/ribs.py` cuts `hollow_interior` (build_sandwich_body's
cavity solid) at each of P3's rib planes (`reference.build_rib_planes`).
Getting a reliable rib from this took 6 rounds of empirical iteration on
`tests/configs/edge/high_taper.yaml` (the hardest available config: 10:1
taper, `mirror:true`, single-ply thin layers — areas from ~12400mm² at the
root down to ~100mm² at the tip) — recorded here because every round
overturned the previous round's working hypothesis, and the FINAL working
route is not the obvious first attempt:

1. **`fuzzy_common(solid, big_plane_face)` to get the cross-section
   directly — WRONG.** A mixed-dimension solid-vs-face boolean is not what
   `BRepAlgoAPI_Common` is meant for; produced `Null TopoDS_Shape` errors at
   3/5 rib planes and invalid/non-watertight results at the other 2.
2. **`BRepAlgoAPI_Section` (the same route `cove_profile.section_points`
   already uses) + `Wire.assembleEdges` — the section/wire-assembly step
   itself is reliable** (verified separately: closed wire, correct edge
   count, at every station, on both the raw loft AND the boolean-restricted
   `hollow_interior` — no difference between them here). The failure is
   further downstream.
3. **Using the raw assembled wire directly for `offset2D`/`extrudeLinear` —
   fragile.** Succeeded at 2/5 stations, and even those "successes" had
   `extrudeLinear` produce an INVALID solid. A many-small-edge wire straight
   from a boolean-result section (435 edges here) is measurably less
   robust for these 2D/extrude operations than a clean, deduplicated
   per-vertex polygon — the same reason `cove_profile.py` already avoids
   feeding raw section output into further construction wherever an
   analytic alternative exists.
4. **Rebuilding as a clean polygon via `edge.startPoint()` per edge — WRONG,
   made it WORSE (0/5).** `assembleEdges`'s output wire can contain
   REVERSED edges; blindly taking `startPoint()` without checking
   orientation scrambles the point sequence into a self-crossing polygon.
   `BRepTools_WireExplorer` (its `CurrentVertex()` respects edge
   orientation while traversing in connection order) is the correct way to
   extract an ordered vertex list from an assembled wire — this fixed
   `offset2D` (0/5 → 5/5 succeeding without raising) but not the
   downstream solid quality.
5. **Two half-thickness `extrudeLinear` calls + `.fuse()` at the shared
   plane — still fragile** (`n_solids` of 0 or 2 instead of a clean 1 at
   several stations): the two prisms share a coincident zero-thickness
   face, the exact F4 tangent-boundary hazard `booleans.py`'s fuzzy
   tolerance exists to avoid — and raw `Shape.fuse()` here was called
   without a fuzzy value. Switched to ONE full-thickness prism from the
   wire translated by `-thickness/2` (no fuse needed at all) — this made
   the plain solid slab (no hole) reliable at 5/5 stations, but the
   HOLE cut (still via `extrudeLinear`'s own inner-wire parameter) stayed
   broken (0/5 clean — `offset2D` "succeeding" without raising was not the
   same as producing a geometrically sane inset on this config; an 8mm
   margin genuinely doesn't fit within a ~100-1200mm² cross-section).
6. **Final route: build the solid slab (step 5, reliable everywhere) then
   cut the lightening hole as a SEPARATE, OVERSIZED prism (3x rib
   thickness) via `fuzzy_cut`** — reusing this codebase's own proven
   fuzzy-boolean helper (booleans.py) instead of `extrudeLinear`'s built-in
   inner-wire handling, the same "generously oversized cutting tool +
   fuzzy boolean" pattern `te_cut.py`'s aft boxes already established.
   VERIFIED, not assumed: every rib is checked for exactly one solid,
   valid, watertight; if the hole cut fails that check (empirically: 4 of
   `high_taper.yaml`'s 5 stations, where the 8mm margin doesn't fit), the
   module falls back to the plain solid slab and records it in
   `RibSet.fallback_solid` — the same graceful-degradation-over-crash
   posture `iml.py`'s own aft self-clip already established for this
   project (D11-style: ramp/skip the optional feature rather than error
   when it doesn't locally fit).

**Verified against the real kernel**: `high_taper.yaml` (standalone probe)
— 5/5 ribs valid + watertight, 4/5 fell back to solid (margin doesn't
fit), 1/5 (the tip, smallest cross-section) got its hole. `te_half.yaml`,
FIRST as a standalone probe (4/7 built, 3 skipped — the 3 rib planes at/in
the device window: y=660/900/1140, matching `span_start_frac`/
`span_end_frac` on the 1200mm span), THEN wired into
`scripts/export_viewer_data.py`'s real pipeline (`build_ribs` called on
`build_sandwich_body`'s actual `hollow_interior`, `include_hollow_interior`
flipped `False→True` since ribs need the cavity solid the viewer-only
shortcut was skipping) and re-run end-to-end: 6/7 built (only y=660, the
device-window START edge, genuinely skipped — 2 fell back to solid, all
6 single/valid/watertight). One rib plane (y=900, inside the window)
built cleanly in the full pipeline where the standalone probe had skipped
it — a specific instance of this project's already-documented
run-to-run variance at marginal/near-degenerate boundary geometry
(docs/known_issues.md), not a construction bug: the SAME code, same
config, same `hollow_interior` boolean can land on either side of a
razor-thin topological margin between runs. y=660 (the device-window
START, where the false spar and cove cut most directly interact) skipped
consistently across both runs — a genuinely disconnected/complex
cross-section there, not run-to-run noise; device-window-edge rib
construction remains the deferred scope limit (module docstring: needs
awareness of the false-spar wall + cove geometry, a follow-on beyond this
increment).

**No new R0 probe beyond what's recorded here** — `BRepAlgoAPI_Section`,
`Wire.assembleEdges`, `BRepTools_WireExplorer`, `Wire.makePolygon`,
`Solid.extrudeLinear`, `offset2D`, and `fuzzy_cut` are all now
empirically characterized for this exact construction (this addendum IS
the probe record; `scripts/r0_probes/probe_ribs_verify.py` re-runs it
against any config).

## Addendum: spars trimmed to IML (plan.md §8.7, 2026-07-12)

`backend/geometry/spar_trim.py` thickens P3's zero-thickness ruled spar
surfaces (`reference.build_spar_surfaces` — a single root-to-tip line per
station, at the spar's interpolated chordwise `xc`) and trims each to the
actual cavity. Much simpler than ribs' R0 trail because it reuses the
SAME per-station-profile-then-ruled-loft pattern every other body in this
project already uses, rather than sectioning a boolean result: per station,
a small rectangle (web-thickness wide, centered on `xc`, generously
oversized vertically past the true `zu`/`zl` — `SPAR_HEIGHT_OVERSIZE_
CHORD_FRAC=0.15`, same role as `false_spar.py`'s `_WALL_HALF_HEIGHT_
CHORD_FRAC`), lofted `ruled=True`, then intersected with `hollow_interior`
via `fuzzy_cut` — same "generously oversized blank + trim with a real
boolean" pattern as ribs' final route and false_spar.py before it. Worked
on the first real-kernel attempt, no dead ends.

One naming change to enable this: `reference.py`'s
`_get_canonical_points_at_xc` (upper/lower skin Z at a given chord
fraction — P3's own spar-surface construction already uses it) is now
PUBLIC (`get_canonical_points_at_xc`, no leading underscore) since
`spar_trim.py` needs the exact same per-station lookup to build its
rectangle at the same `xc` P3's zero-thickness line uses. No other call
sites existed to break.

**Verified against the real kernel**: `high_taper.yaml` (1 spar) — single
valid watertight solid, vol=9775.8mm³. `te_half.yaml` (2 spars, full
pipeline including the device window) — `main` vol=33811.4mm³ (1 solid),
`rear` vol=8771.8mm³ split into 2 watertight solids. The rear spar's split
is PHYSICALLY EXPECTED, not a defect: at `xc=0.70` (close to the device's
`hinge_xc=0.75`), the wing-side rear spar's own web naturally discontinues
where the hinge/false-spar mechanism takes over structurally — the ribs.py
`RibSet.skipped_no_section` device-window scope limit applies to the same
underlying geometry. `build_trimmed_spars`' verification (in
`scripts/export_viewer_data.py`) accepts any number of ≥1 valid/watertight
solids per spar rather than requiring exactly one, for this reason.

## Addendum: midsurface faces (plan.md §8.7, D15, 2026-07-12)

`backend/geometry/midsurface.py` builds the wing-skin midsurface; ribs and
spars already had theirs as free byproducts of work already done this
phase (ribs.py's own pre-thickening wire is now exposed as
`Rib.midsurface_face`; `reference.build_spar_surfaces`'s zero-thickness P3
ruled shells ARE the spar midsurfaces exactly — spar_trim.py only added
thickness+trim on top). D15 ("Mechanical layered shell sections") settles
what "structural body count" means for the P6 gate's own pass criterion:
ONE shell per WALL, not one per sandwich LAYER — the face/core/face layup
is a shell-section MATERIAL property applied later with the layup
schedule export (a future phase), not multiple separate midsurface
geometries per layer. So: 1 skin midsurface (not 6, matching the 6
upper/lower solid shells) + 1 per rib + 1 per spar.

**Genuinely new API surface, R0-probed first**
(`scripts/r0_probes/probe_ocp_shell_loft.py`): a midsurface must be an
OPEN shell (no enclosed volume), but `cq.Solid.makeLoft` — used
everywhere else in this project — hardcodes `BRepOffsetAPI_ThruSections(
isSolid=True, ...)` (its own source comment: "the True flag requests
building a solid instead of a shell"). Calling the SAME underlying OCC
constructor directly with `isSolid=False` is the other documented mode of
an already-proven API, not a new one. Verified on the real kernel before
use: face count and total area of the `isSolid=False` shell loft exactly
matched the equivalent solid loft's LATERAL faces (excluding its 2 end
caps) on a live `te_half.yaml` section — 199 faces / 883024.4 mm² both
ways.

**Clean-span only** (module docstring, same scope limit `iml.py` itself
started with): the skin midsurface uses the plain per-station offset at
half the panel's total stack thickness from the OML, with NO ramp or
cove-fidelity correction applied. An explicit, tracked follow-on — the
same posture as every other device-region refinement already deferred
this phase (CS-nose sandwich, hinge-land/joint/hardpoint ramping,
device-window-edge ribs).

Two more private→public promotions this addendum needed (same pattern as
`get_canonical_points_at_xc` for spar_trim.py): `iml.offset_wire` (was
`_offset_wire`) — midsurface.py reuses it for the half-stack offset.

**Verified against the real kernel**: `high_taper.yaml` — skin (338
faces, area=447074.4mm², 0 solids), 5 rib + 1 spar midsurfaces, all valid;
7 midsurfaces == 7 structural bodies (1 skin + 5 ribs + 1 spar). Full
pipeline (`scripts/export_viewer_data.py`) on `te_half.yaml`, WITH ribs
and trimmed spars: skin area=848403.6mm², 9 midsurfaces == 9 structural
bodies (1 skin + 6 ribs + 2 spars, matching that run's own rib/spar
counts). 0.9s added cost — negligible next to the sandwich-shell booleans.
