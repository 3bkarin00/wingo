# R0 Findings — P1 (Airfoil subsystem)

The P1 "third-party boundary" is the **UIUC .dat file format itself** (there
is no library that parses it for us — F13 applies: understand the real data
before writing the parser). Hand-analyzed 8 real files fetched from
`https://m-selig.ae.illinois.edu/ads/coord/<name>.dat`.

## The two formats (confirmed from real files)

### Selig — single continuous loop, TE → upper → LE → lower → TE
`s1223`, `ag24`, `e387`, `naca2412`, `fx76mp140`, `sd7037`.
```
S1223                <- name/header line (line 1)
  1.00000  0.00000   <- first coord is the TRAILING EDGE, x = 1.0
  0.99838  0.00126
  ...                <- upper surface, x decreasing toward LE
  0.00000  0.00000   <- leading edge
  ...                <- lower surface, x increasing back toward TE
  1.00000 -0.00000   <- back to TE
```
This is already our canonical order (conventions.md §airfoils). Parse as-is.

### Lednicer — two blocks, each LE → TE, preceded by a counts line
`clarky` (`61. 61.`), `goe398` (`17. 17.`).
```
GOE 398 AIRFOIL      <- name/header line (line 1)
      17.      17.    <- COUNTS line: n_upper n_lower (floats w/ trailing dot)
                      <- blank line
 0.0000000 0.0000000  <- upper block, LE -> TE (x increasing)
 ...
 1.0000000 0.0000000
                      <- blank line
 0.0000000 0.0000000  <- lower block, LE -> TE (x increasing)
 ...
 1.0000000 0.0000000
```
Convert to canonical Selig order: `reverse(upper)` (gives TE→LE) then append
`lower[1:]` (drop the duplicated LE point) → TE→upper→LE→lower→TE.

## Detection heuristic (the decision this R0 exists to lock in)

1. Line 1 is always a name/header (may contain spaces, `|`, and a long
   description — see `ag24`). Never parse it as coordinates.
2. Scan for the first subsequent line that parses as two floats `(a, b)`.
3. **If `a > 1.5` → Lednicer** — `a` is a point count. Corroborating signals
   (not required): `a` and `b` near-equal, both integral, both ≥ ~5.
4. **Else → Selig** — that line is the first coordinate (the TE at x≈1.0).

**Edge case (found via the P1 smoke test, threshold revised from 1.0→1.5):**
published Selig data can round the TE x-coordinate marginally OVER 1.0 —
`naca23012.dat` begins at `x = 1.00003`. A threshold at 1.0 misreads that as a
Lednicer count of "1" and crashes on the empty lower block. The real gap is
wide: Selig first-x ≤ ~1.001, Lednicer counts ≥ 5 (snapshot min = 17), so
**1.5** sits safely in the empty middle. Confirmed on all 30 vendored files
(12 Lednicer counts: 17/31/42/49/61; 18 Selig first-x ≤ 1.00003).

## Real-world messiness to handle (else quarantine with a reason string)

- **CRLF line endings** (`goe398` has `\r\n`) — strip `\r`.
- **Leading-dot numbers** (`-.0121500`, `.0343500`) — Python `float()` parses
  these fine; a naive regex might not.
- **Blank lines** between/around blocks — skip them.
- **Trailing-dot counts** (`17.`, `61.0`) — `float()` handles; round to int.
- Quarantine (do NOT silently drop — gate requires 0 silent failures) when:
  a line can't be parsed, a Lednicer file's block lengths don't match its
  declared counts, point count is implausibly low, or the loop isn't closed /
  can't be normalized.

## NACA generator reference (for the "match published ordinates" gate)

NACA 4-digit thickness (already in `backend/airfoils/naca_thickness.py`) and
5-digit camber use the standard closed-form equations; the gate checks 3
reference sections against published ordinate tables (e.g. NACA Report 460 /
Abbott & von Doenhoff appendix) to a documented tolerance. Expected ordinate
values are committed under `tests/golden/expected/` with provenance, not
hardcoded as magic numbers (§0.2).
