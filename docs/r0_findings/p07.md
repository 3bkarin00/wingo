# R0 Findings — P7 (Hinges, generated mode)

## probe_hinges_geometry.py (round 1) — where does material actually exist near the hinge axis?

Real config: `tests/configs/devices/te_half.yaml` (gap_mm=5.0, hinges: {mode: generated, count: 3}).
Real build: OML -> P4 device cut (`cut_te_surface`) -> P6 false spar (`build_false_spar`).

- Tiny (0.1mm) probe box centered exactly on the true hinge axis, at the
  device window's midspan station: **100% inside `cs_solid`, 0% inside
  `wing_solid`**. CS's nose material reaches all the way to r=0 (the axis
  line itself); wing has ZERO material anywhere near the hinge axis within
  the device window.
- A candidate knuckle (OD=5mm cylinder, centered on the axis) is **100%
  inside `cs_solid`** at that same station — confirms a TANG (CS-side
  hardware) can be built directly as `fuzzy_common(knuckle_blank, cs_solid)`,
  the same "oversized blank -> boolean -> real trimmed body" idiom
  spar_trim.py already established.
- Cutting a candidate LUG blank (knuckle + a reaching tab toward the false
  spar) against `te_cut.build_cove_offset_region(extra_radius_mm=0.0)` (the
  same real function P6 already uses for cove-fidelity IML cuts) does NOT
  merely trim the tab's near-axis sliver — it removes the entire near-axis
  region: lug volume before/after = 2196.35mm³ -> 1254.75mm³, i.e. the whole
  196.35mm³ knuckle plus part of the tab is gone. Measured clearance to
  `cs_solid` after the cut: 4.9946mm — NOT >= gap_mm=5.0 on this exact
  config (a coincidence of gap_mm == COVE_CLEARANCE_MM here, but the
  underlying construction gives ~COVE_CLEARANCE_MM regardless of gap_mm,
  which is wrong in general).
- CONCLUSION: a wing-side lug CANNOT be built as a subtraction result at
  all — CS's real material occupies the near-axis space a naive "cut
  against cs_solid/cove_region" approach assumes is free.

## probe_hinges_geometry2.py / probe_hinges_geometry3.py (rounds 2-3) — the notch-tool fix, and why the first version of it failed

Fix hypothesis: instead of subtracting from the lug's own blank, carve a
GROWN "keyway" copy of the lug's own shape out of a DERIVED copy of
`cs_solid` ("cs_notched" — never the frozen P4 `cs_solid` itself), then
build the lug as the un-grown shape.

Round 2 (grow each construction PARAMETER — knuckle radius, box
dimensions — by `margin = gap_mm + 0.05`) measured
`lug_to_cs_notched clearance = 0.0` — NOT the expected `margin`. Round 3
isolated why: `lug ∩ notch_tool` volume was only 512.674mm³ of the lug's
own 1090.563mm³ (47% containment) — growing each fused-shape's PARAMETERS
independently does not guarantee the resulting fused (cylinder ∪ box)
shape is a superset of the original grown by margin everywhere. The
zero-distance point (local a,u,h = -10.085, -2.5, -5.0 relative to the
lug's own center) sits almost exactly on the nose arc's true circular
boundary (`sqrt(a²+u²) = sqrt(10.085²+2.5²) = 10.39 ≈ measured feet.R =
10.384`) — a box corner effect combined with the real loft being a RULED
interpolation between discrete stations (not a perfect circle at every
intermediate spanwise position), not captured by a naive per-parameter
scale-up.

## probe_hinges_geometry4.py (round 4) — the robust fix, confirmed

Fix: grow the lug's own real `BoundingBox()` by `margin` in **every global
X/Y/Z face direction** (not the local a/u/h frame, not per-construction-
parameter) to build the notch tool. This is a provably-correct Minkowski-
box guarantee independent of the lug's actual shape: for any point p inside
the original bounding box and any point q outside the grown box, at least
one axis has `|p_i - q_i| > margin`, so `|p-q| > margin` unconditionally.

Measured on the real kernel, same config/station:
- `lug ∩ notch_tool` volume = 1090.563mm³ = **exactly** `lug.Volume()`
  (100% containment, vs round 3's 47%).
- `cs_notched`: 1 solid, 0 shards, watertight=True.
- `lug_to_cs_notched clearance = 5.049999999999983mm` — **exactly**
  `margin` (gap_mm=5.0 + 0.05mm safety term), i.e. >= gap_mm as required.
- `lug_to_false_spar ≈ 4.44e-16` (touching) with `lug ∩ false_spar` real
  overlap volume = 0.264mm³ (positive — a genuine bond, not a zero-distance
  F4-adjacent touch).
- `tang` (built the round-1 way, `fuzzy_common(knuckle, cs_solid)`):
  volume=196.35mm³ (100% of the knuckle blank), 1 solid, 0 shards,
  watertight=True; `tang_to_wing` clearance = 12.79mm (comfortably >=
  gap_mm=5.0, expected since wing has no material anywhere nearby).
- After drilling coaxial pin bores (fuzzy_cut with a cylinder centered
  exactly on the true axis point, same base/dir as the knuckle itself):
  both `lug_drilled` and `tang_drilled` remain 1 solid, 0 shards,
  watertight. Extracted cylindrical faces (both the knuckle's OUTER wall,
  radius=2.5mm, and the pin bore's inner wall, radius=1.0mm) report
  **axis_dev=0.00000mm** against the true hinge axis line for both lug and
  tang — comfortably inside `COAXIALITY_TOLERANCE_MM=0.05mm`.

This is the construction `backend/geometry/hinges.py` implements. No new
third-party API beyond what P4/P6 already R0-verified (`cq.Solid.
makeCylinder`, `cq.Solid.makeBox`, `.BoundingBox()` — all already used
elsewhere in this codebase, e.g. `false_spar.py`/`export_viewer_data.py`).

## probe_rotation_axis.py — CORRECTION to the first run above

First run reported both rotation techniques FAILING an analytic accuracy
check by 1.125mm. Root cause found by isolating pre-rotation vs
post-rotation state: `cq.Solid.makeSphere(radius, cq.Vector(*center))`
(2-arg call) does NOT produce a full sphere centered at `center` in the
installed cadquery version — its default angle parameters produce a
HEMISPHERE whose centroid sits 3/8*radius above `center` (1.125mm for the
probe's r=3mm sphere, matching the "deviation" exactly). This was a bug in
the probe's own assumption about makeSphere's defaults, not a rotation API
bug — a good illustration of why R0 probing checks the WHOLE chain
including setup, not just the API under test.

Re-run with an EXPLICIT full sphere
(`makeSphere(r, pnt, cq.Vector(0,0,1), -90, 90, 360)`): pre-rotation
Center() now matches the intended center exactly, and post-rotation
deviation from the analytic Rodrigues-formula reference is 1.5e-14mm
(floating-point noise) for both `cq.Shape.rotate(p0, p1, angle_deg)` and
raw `gp_Trsf().SetRotation(gp_Ax1(...), radians) + BRepBuilderAPI_Transform`.
Zero-angle and axis-direction-sign checks also passed (0deg leaves the
center exactly unchanged; 25deg p0->p1 exactly equals -25deg p1->p0).

**CONCLUSION (supersedes the first run's FAIL verdict): `cq.Shape.rotate(axisStartPoint, axisEndPoint, angleDegrees)`
is accurate and is the technique to use for WP1's union-of-rotated-copies
swept-pocket construction and for P8's kinematic sweep — both consume an
arbitrary 3D hinge-axis line exactly like this probe's setup. Caution for
future probes/code in this codebase: `cq.Solid.makeSphere`'s 2-arg call
is NOT a full sphere in this environment; always pass explicit angle
arguments when a true full sphere is needed.**
